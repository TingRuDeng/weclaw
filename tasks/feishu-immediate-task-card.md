# 飞书即时任务卡片设计

## 状态

已确认，等待实施计划审查。

## 目标

飞书收到 Codex 或 Claude 任务后立即创建一张任务卡片，执行期间只更新该卡片的最新结构化状态，任务结束时把完整回答一次写入同一卡片，并额外发送一条简短通知。

## 非目标

- 不把 Agent 最终回答的 token 或正文 delta 逐字写入卡片。
- 不改变微信的 typing、进度或最终回复行为。
- 不增加新的用户配置字段或进度模式。
- 不修改 Codex、Claude 的会话、模型、审批和任务队列语义。

## 当前行为与问题

- `messaging.progressSession.start` 只启动进度循环，不立即调用 `OpenStream`。
- 默认 `initial_delay_seconds` 为 10 秒，默认汇总间隔为 20 秒；没有后续 delta 时，飞书首张任务卡片可能约 20 秒后才出现。
- 当前飞书声明 `FinalReplyOutsideStream`，任务卡只展示状态，完整回答作为独立消息发送。
- Codex 与 Claude 都通过 `messaging.startProgressSessionWithFinal` 接入进度会话，因此应在消息层统一修复，不建立 Agent 专属卡片实现。

## 用户体验

### 启动

- `summary`、`verbose`、`stream`、`debug` 模式在任务登记后立即创建飞书任务卡片。
- 卡片标题使用现有任务标题截断逻辑。
- 初始状态为“处理中”，正文统一显示“正在处理任务，请稍候。”，不为展示 Agent 名称扩大进度会话参数。
- `off` 和 `typing` 模式保持现有行为，不创建任务卡片。

### 执行中

- 每次只展示最新一条结构化进度，不保留历史日志。
- 沿用现有去重、最大消息数和 CardKit 500ms 更新节流。
- Agent 最终回答正文不作为进度内容，避免半截 Markdown、闪烁和重复。

### 成功

- 同一卡片切换为完成状态，正文替换为完整最终回答。
- 卡片更新成功后发送独立文本：`任务已完成，请查看上方卡片。`
- 独立通知不重复最终回答正文。

### 失败与取消

- 同一卡片切换为失败或停止状态，正文显示现有友好错误或停止原因。
- 失败卡片更新成功后发送独立文本：`任务执行失败，请查看上方卡片。`
- 停止卡片更新成功后发送独立文本：`任务已停止，请查看上方卡片。`

## 状态与数据流

1. `messaging` 解析账号级、平台级和 Agent 级进度配置。
2. `progressSession.start` 判断平台支持原生流式卡片且模式允许进度后，立即打开 stream，并写入初始内容。
3. Codex 或 Claude 的结构化进度进入 `onProgress`，经现有渲染、去重和节流后更新同一 stream。
4. Agent 返回最终结果后，消息层先生成交付投影，把可展示正文与本地附件、图片分离，卡片正文不得暴露内部附件标记。
5. 独立媒体发送结果写回可展示正文后，`finishProgressWithReplyForPlatform` 尝试把完整正文写入 stream 终态。
6. stream 终态成功时返回 `consumed=true`，消息层只发送简短完成通知；附件和图片已通过独立消息交付。
7. stream 终态失败时返回 `consumed=false`，消息层发送完整普通文本，不再发送误导性的完成通知。

## 平台能力

- 在 `platform.Capabilities` 增加“stream 终态成功后需要独立通知”的能力标记。
- 飞书关闭 `FinalReplyOutsideStream`，启用新的完成通知能力。
- 微信和其他 Replier 不启用该能力，行为保持不变。
- 通知由通用消息层根据能力和真实终态发送，不在 Codex、Claude 或飞书业务入口重复实现。

## 错误处理

- 初始卡片创建失败时，显式发送：`任务已开始，卡片创建失败，将以普通消息返回结果。`
- 卡片创建失败不终止 Agent 任务；最终结果继续走普通消息。
- 最终卡片更新失败时，不发送“请查看上方卡片”，而是发送完整普通文本结果。
- 简短通知发送失败只记录明确日志；卡片已经保存权威最终结果，不重复发送完整正文。
- 最终回答包含本地附件或图片时，消息层先发送独立媒体，再把发送结果写入卡片正文；无法安全放入卡片的文本继续使用现有普通消息路径。

## 配置兼容

- 不增加配置字段。
- 飞书已有 `summary`、`verbose`、`stream`、`debug` 配置自动获得即时卡片行为。
- `/progress off` 和 `/progress typing` 明确禁用任务卡片。
- 多机器人仍使用现有账号级进度配置解析，不改变覆盖优先级。

## 测试要求

- RED：启动进度会话后、尚无 Agent delta 时，飞书原生 stream 已创建且包含初始状态。
- RED：短任务把完整结果写入已创建卡片，并只额外发送简短完成通知。
- RED：成功、失败、取消分别产生正确卡片终态和通知文案。
- RED：卡片创建失败时发送明确提示，最终完整结果走普通文本。
- RED：卡片终态更新失败时不发送完成通知，完整结果不丢失。
- RED：`off`、`typing` 和微信路径不创建即时任务卡片。
- RED：Codex 后台任务、Claude 同步任务和外部 Codex watcher 使用相同终态规则。
- 保留并更新审批卡片、附件回传、任务队列和 CardKit 节流相关回归测试。

## 验收标准

- 飞书发送 Codex 或 Claude 消息后，无需等待 Agent 首个事件即可看到任务卡片。
- 执行期间聊天中只有一张任务卡，卡片只展示最新状态。
- 成功后完整回答位于任务卡，聊天中另有且仅有一条简短完成通知。
- 卡片不可用时用户仍能收到完整普通文本结果，并能看到明确的降级原因。
- 微信行为与变更前一致。
- `go test ./...`、`go test -race ./...`、`go vet ./...`、`staticcheck ./...`、文档校验和 `git diff --check` 全部通过。
